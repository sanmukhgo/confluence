Inventory Storage

Flexible storage structure

Overview
  • Inventory storage can model any rack and its composition in a cuboid

  • The rack-type grammar includes flexible hierarchy of rackÂ â†’ floorÂ â†’ base â†’Â faceÂ â†’ slots

  • The grammar can support multiple storage racks such as MPU (mobile pallet unit) and MSU (mobile storage unit)

  • Every rack is in instance of a certain rack type. Rack-types are created as a JSON structure and are part of installation

  • There is no restriction of a floor to extend to the entire MSU base i.e. one can create partial floors in sub-slotting a slot again
  • The final storage entity "slot" has three types
    • Slot: normal cuboid slots wherein normal storage algorithm will be used
    • Hanger: normal cuboid slots except that stability check is disabled
    • Drawer: unique type of slots wherein gravity and open face are different from the front face of the rack
Limitations
  • All the levels of storage have to be cuboid mapped i.e they need to have a length, depth, height
  • Floor is assumed to be planer with respect to the rack base
  • The down-stream system(s) assume two faces only in a rack storage
  • Except for using set of scripts, racks cannot be dynamically corrected using an updated rack-type. It's advisable to not run any functions other thanÂ scripts from the team

Inventory storage

Overview
  • Inventory storage only supports storage for butler inventory i.e. inventory which can be stored in MSUs or racks
  • Inventory storage itself doesn't restrict any type of product as long as it's cuboid
  • It supports n-level of mixed containing units i.e. it can model different package UOMs with different quantities and inventory carrying units such as totes
  • Storage itself can support storing serialized and non-serialized (with multiple barcodes on same product) with different UOMs
  • The serials are tracked separately with it's own history at inventory tracker
  • A hanger slot only supports storing inventory that is of the hanger type
Limitations
  • Non-cuboid objects have to be stored as a cuboid. It potentially might waste space for e.g. hockey sticks and dustbins
  • While database models do not impose a restriction on storing heterogeneous hierarchal inventory, put processor and process flow don't support it
  • There is no stock ledger maintained for non-serialized inventory

Storage algorithms

2-D space storage with depth (re)-fill

Overview
  • The goal of storage algorithm to minimize the probability of operator not been able to put a product if the system says that it has space
  • In other words, if system points that there is available space, there should exist at-least one way for the operator to fit items even if by re-arranging
  • The frontal face area is carved out as a free space and it's packed with an unique face out of the three possible faces of an product orientation
  • Free space algorithm ensures that the system knows about largest disjoint contiguous set of free spaces per every slot
  • While fitting an item, for a simple product (with no storage preference)
    • The face that is selected ensures that the minimum area is taken out of the free space
    • The product is ensured that it's a stable configuration. The height cannot be more than 3 times any other dimension
  • When an item is picked, gravity ensures that the items are pushed into the gravity direction in aÂ 2-D area. Free spaces are re-calculated so that consolidation of free spaces can be done
  • When putting multiple quantities of same product, depth is full reserved till the depth of the slot
  • During packing, item bottom's left corner is aligned with the free space corner
  • During the same put, multiple quantitiesÂ are packed behind the front item to utilize depth . If slot re-fill isÂ 
    • On: during further put and picks, this depth can be re-utilised for the product and it's UOM if space is available
    • Off: depth is released only when all items are picked out

Volumetric storage

Overview
  • Certain type of puts require only simple storage algorithm such as user directed put.Â 
  • The volume is calculated for product, UOM and it's quantity and is filtered at the free volume left
  • Given that the same database tables get used for all the algorithms, items are considered at the bottom left of the slot i.e (0, 0, 0)
  • The volume is immediately freed as soon as items are picked i.e there is no further blocked reservation

Storage strategy

Overview
  • Storage strategy is used to direct a staged product to a certain type of slot.Â A slot already has a storage tag assigned to it during racktype installation
  • When a product and it's UOM is put, it has a set of storage tags in a order preference, which is fetched from storage strategy in MDM
  • All the slots are searched for available space in the order of preference
  • We associate a tag matching with item and it's package UOM combination so outer and inner can effectively go to different slots based on their strategies
  • Hanger inventory is already a slot preference with the nature of it's inventory
Limitations
  • Currently, storage strategy only works with system directed put i.e it doesn't work with user directed put and pallet put
  • There is no interface to change the slot tags for a slot. Storage strategy for products can however be changed with MDM APIs

Exclusions

Overview
  • Exclusion involves ensuring that based on a configured rule type, products are excluded either with a location or with certain family or products
  • These rules are relevant when products of one seller cannot be with another seller or certain categories cannot be together (perfumes with food)
  • Exclusion location can be supported at rack, rack-face, shelf or slot level
  • The exclusions rules are made at MDM and they are posted with the product information
  • There are two type of exclusion rules
Rule typeNotes
HardHard exclusion ensures that if rule fails, then the product is not put inside the warehouse
SoftSoft exclusion gets invoked when product is tried with exclusion, and if a rule fails, system tries to find a slot without exclusion. This would ensure maximum products are kept with exclusion rules but not discard put

Limitations
  • The containers are assumed to be homogeneous i.e the outer have to contain the same SKU and same UOM
  • Currently, storage strategy only works with system directed put i.e it doesn't work with user directed put and pallet put
Configurations
  • Exclusions have to be configured in racktypes and MDM with a fair bit of rule engine APIs

Slot product limit

Overview
  • There is a global product limit to ensure that only item comes for a particular product
  • Having a small product limit ensures better search time and limited scope for errors during put/pick operations
  • This feature typically takes the value of 1 as most clients transit from manual warehouse put/pick
  • Multiple UOMs of the same product can come in one slot
Limitations
  • Currently, slot product limit only works as a system wide flag
  • Having a small limit will decrease the storage utilisation and there is no way to estimate that
  • We theoretically can improve storage utilisation with dynamic slot product limits in the future
  • Slot product limit only applies for system directed put. It's not applicable for user directed and pallet put as they by pass system direction

Non-functional put processing

Center of gravity

Overview
  • CG calculations are done during put transaction to ensure that the rack stability is maintained while the rack movement is there
  • Bot goes through multiple movements: acceleration, de-acceleration, steady speed, turn, E-pause, E-stop
  • It is assumed basis the calculations from mechanical teams that worst case CG limits are in acceleration and de-acceleration.Â 
  • Both of these limits are same so ensuring one set of limits ensure that rack is safe in all movements
  • Center of gravity calculations get factored in MPU or MSU, however at MPU CG is assumed to be at (0,0,z) as we assume to be balanced pallet to be inducted
  • Non-centered bot suspension introduces asymmetricÂ limits, however so far we assume the worst of the both and symmetrical limits
  • Basis the limits given by mechanical teams, we run regression to feed the variables for the CG cone
  • The system tries to do the maximum put on a rack and then selects the next best rack
  • During CG calculations, mass limit of rack and shelf limit of rack are also respected since they are bot limit and physical constraints of MSU

  • CG calculations are done using two methods

    Solution typeDetails
    Linear programmingDeterministic solution to calculate the quantity distribution. Uses a third part LP solver via a C++ program (coinlp)
    Approximation heuristicsUses simulated annealing to get to the best solution possible. Is significantly fast compared to linear programming and reasonably accurate
Limitations
  • CG calculations are only supported for system directed put in MSU i.e it's not supported for user directed put or post-pack operations
  • Current limits only support single bot installation and MSU 2.0
  • Given CG implementation, forced fragmentation of inventory is not supported with any metric
  • We don't assume any suspension/non-suspension side of the bot picking up the rack
  • The system directed codebase has branched into two put calculation of
    • With CG constraint
    • Without CG constraint

Order Fulfilment Engine

Multi-level order modelling

Overview
  • Order modelling supports infinite tree hierarchy for pick orders.Â 
  • Singular structure mapping order and orderlines thus acting a single storage of an incoming and/or processed order
  • Supports multiple hierarchy orders wherein a client order(s) goes through mutation to become an eligible order for order processing. The final eligible order processing can be further mutated during processing lifecycle, however such an use case has not been seen yet
  • The order structure has relevant methods to navigate through the tree: children(N), siblings(N), parent(N)
  • All the processable orders (i.e. eligible for order processing) are indexed at DB layer.Â 
  • An pickable order needs to have a valid pick location type. Currently only two types are supported
    • Rollcage
    • Bin
  • Bin pick location type cannot be broken down to any smaller units. Bin is the smallest unit
Limitations
  • Single mutation has been implemented however, it should not be complicated to support multiple mutations of an upstream order
  • Given the implemented support in process flows, the validÂ pick location type hierarchies

Order pre-processing

Pick after time

Overview
  • Pick after time can be provided for an order. System converts this time to UTC
  • System maintains a queue in order manager. At the time of initialisation, the system boots up and prepares callbacks with the time delay
  • There is no soft pick after time. System will not process pick after time orders until their time has arrived
Limitations
  • We don't reject orders based on the capacity. Orders can be breached

Pick before time

Overview
  • Orders can be batched or posted in real time by the client. System exposes a PBT field to ensure that the order SLAs are met well
  • The orders given by a client will have batched orders or discrete orders or a mixture of both. In addition, orders can come in advance or be given in real time.
  • System convert the received PBT time to UTC internally
  • System maintains pick required per every discrete PBT window. It uses intelligent strategies to combine or prioritise near-PBT orders depending on the pick capacity calculated
  • Pick capacity is dynamically calculated with the number of picks happening in a certain window multiplied with active PPS
  • If any order is breached, the breached orders are prioritisedÂ in the pre-selection in terms of their ageingÂ cost

  • System provides two strategies

    StrategyGoal
    Best effort PBT

    The strategy aims at optimizing throughput for the day. PBT specified for orders is not a hard constraint. Throughput is never compromised to fulfil orders whose PBT is approaching.

    Orders are combined based on their overall priority. Orders can be combined across PBTs and with orders without a PBTs. Order combining is never stopped even when a deadline is approaching. This ensures that throughput of system is never compromised to pick orders of nearest deadline.

    Strict PBT

    The strategy tries to minimize breach for orders. Best possible throughput may not be achieved as order combining is restricted for certain time durations.

    Orders with different PBTs are combined based on their overall priority and cost. Orders with and without PBT will also be combined based on their overall priority.When the nearest pick before time is approaching, order combining is reduced. When the current capacity of system is equal to number of pending orders for nearest PBT, order combining is stopped. At this point, orders with different PBTs are not combined. Orders with PBT are not combined with orders whose PBT is not specified.This might reduce the overall throughput of the system.

Limitations
  • There are no breach policies implemented in the system
  • The system finally selects an order which is an ingredient of multiple heuristics. While PBT might become an empowering factor, there might be other factors which might overpower if the weights are not set right or warehouse conditions prefer some other order

Client order priority

OverviewÂ 
  • Client can provide order priority to influence the order processing and task selection during picking
  • Pre-processing adds a simple cost to the overall priority of an order for it to be selected for order processing
  • Client can provide either provide numeric value for an priority or a simple priority field: high, medium, low, critical
  • Critical orders are special orders in the sense they will always get sorted at the top and then processed as soon as possible for task creation
  • The two orders fields can be used. If both are present simple priority is prioritised
    • simple_priority
    • order_priority
  • Order priority can be modified externally using a modification API. Order priority modification can happen at any stage of the order
  • Order priority is linear in the calculations and a lower order priority is better
Limitations
  • System will not be able to prioritise well if most of the orders are critical. A well balanced mix has to be decided well in the solutioning phase
  • If the weights are not set right, then there is a chance the client order priority might not be a dominant factor in order selection
